# Bugfix 工作流

精简的缺陷修复流程。快速定位→修复→验证。

## 流程

```
用户: "有个bug" / "报错了"
  │
  ├─ ► 阶段1: 问题定位 [子Agent: debugging]
  │     收集信息→启动子Agent定位→得到根因+修复建议
  │     决策: 根因明确 → 阶段2
  │
  ├─ ► 阶段2: 修复实现 [子Agent: code-implementation]
  │     写测试复现→修复代码→测试通过→回归测试
  │     决策: 全部测试通过 → 阶段3
  │
  └─ ► 阶段3: 验收 [主Agent]
        修复报告: 根因+修改+测试结果
```

---

## 阶段1: 问题定位

### 主Agent动作

1. 收集用户报错信息
2. 评估严重程度
3. 启动子Agent: `debugging`

### 严重程度评估模板

```markdown
## 严重程度

| 级别 | 标准 | 示例 |
|------|------|------|
| P0 | 核心功能不可用，影响所有用户 | 登录失败、支付崩溃 |
| P1 | 功能异常但有 workaround | 列表加载慢、图片不显示 |
| P2 | 体验问题，不影响主流程 | 动画卡顿、文案错误 |
| P3 | UI细节，低影响 | 间距偏差、颜色差异 |
```

### 信息收集清单

```markdown
## Bug信息收集

### 用户侧
- [ ] 操作步骤（1→2→3）
- [ ] 期望结果
- [ ] 实际结果
- [ ] 截图/录屏
- [ ] 设备型号 + 微信版本
- [ ] 复现频率（每次/偶尔/特定条件）

### 技术侧
- [ ] 错误堆栈/日志
- [ ] Network请求日志
- [ ] 最近相关代码变更
- [ ] 是否与特定数据相关
```

### 传递子Agent

```
task: "debugging"
projectPath: [项目路径]
scope: [报错涉及的文件]
context: { errorInfo: "用户描述+堆栈+复现步骤" }
```

---

## 阶段2: 修复实现

### 主Agent动作

1. 启动子Agent: `code-implementation` (修复模式)
2. 传递: `{ task: "code-implementation", projectPath, scope: affectedFiles, context: { fixSuggestion: rootCause.fixSuggestion } }`

### 修复验证流程

```
1. 编写复现测试 → 测试失败（确认复现bug）
2. 修改代码 → 测试通过（确认bug修复）
3. 运行相关回归测试 → 全部通过（确认无副作用）
```

---

## 阶段3: 验收

### 验证检查清单

```markdown
## 修复验证

### 直接验证
- [ ] 原始复现步骤不可复现
- [ ] 新增测试用例通过
- [ ] 相关UT全部通过

### 回归验证
- [ ] 受影响模块的测试通过
- [ ] 上下游依赖模块测试通过
- [ ] 相关E2E场景通过

### 边界验证
- [ ] 修复场景的边界值测试
- [ ] 同类问题排查（其他模块是否有同样模式）
```

### 验收报告

```markdown
# Bugfix 完成报告

## Bug: [描述]
## 严重程度: [P0/P1/P2/P3]
## 修复耗时: X分钟

## 根因
- 类型: [逻辑错误/数据问题/环境配置/权限]
- 文件: [路径+行号]
- 原因: [一句话]

## 修复内容
- 修改文件: X个
- 新增测试: X个（复现bug的测试 + 边界测试）

## 测试结果
| 类型 | 通过 | 失败 |
|------|------|------|
| 新增测试 | X | 0 |
| 回归测试 | X | 0 |

## 同类排查
- [ ] 已排查其他模块是否有同类问题
- 存在同类问题的文件: [列表/无]

## 状态: ✅ Bugfix 完成
```
