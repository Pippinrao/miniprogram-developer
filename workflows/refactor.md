# 重构工作流

安全的重构流程，核心是先建保护网再动手。

## 流程

```
用户: "重构XXX" / "代码太乱了"
  │
  ├─ ► 阶段1: 重构规划 [主Agent]
  │     输出: 重构范围、影响分析、重构策略
  │     决策: 用户确认 → 阶段2
  │
  ├─ ► 阶段2: 保护网建设 [子Agent: test-execution]
  │     分析现有测试→补充缺失UT→运行基线→记录覆盖率
  │     决策: 基线测试全部通过 → 阶段3
  │
  ├─ ► 阶段3: 逐步重构 [子Agent: refactoring]
  │     小步提交（≤10行/次）→ 运行测试 → 全部通过 → 提交
  │     决策: 重构完成 → 阶段4
  │
  └─ ► 阶段4: 验收 [主Agent]
        覆盖率对比 + 功能完整性 + 验收报告
```

---

## 阶段1: 重构规划

### 主Agent动作

1. 理解用户重构目标
2. 输出重构规划

```markdown
# 重构规划: [目标]

## 重构动机
- [ ] 代码难以维护（重复代码、长函数、耦合高）
- [ ] 性能问题（渲染慢、内存泄漏）
- [ ] 架构升级（框架版本、设计模式）
- [ ] 技术债务清理

## 重构范围
### 纳入
- 文件/模块: [列表]
- 预估改动: XX行

### 排除（明确不碰）
- [列表，含原因]

## 影响分析
| 被依赖方 | 影响 | 缓解 |
|----------|------|------|
| 页面A | 高 | E2E回归 |

## 重构策略
- [ ] 大拆小: 分 N 步，每步独立提交
- [ ] 并行: 独立模块同时重构
- [ ] 一次性: 小范围快速完成

## 预计工作量
[X]步 × [Y]min/步 = [Z]分钟
```

3. **必须等用户确认**

---

## 阶段2: 保护网建设

分三步建立重构安全网:

### 步骤1: 评估现有覆盖 [子Agent: code-analysis]

1. 启动子Agent: `code-analysis`
2. 传递: `{ task: "code-analysis", projectPath, scope: "重构范围" }`
3. 从输出中获取 `testability` 和现有测试覆盖情况

### 步骤2: 设计缺失测试 [子Agent: test-design]

1. 启动子Agent: `test-design`
2. 传递: `{ task: "test-design", projectPath, scope: "重构范围", context: { analysisResult } }`
3. 获取 `utCases` 列表

### 步骤3: 执行并记录基线 [子Agent: test-execution]

1. 启动子Agent: `test-execution`, mode=ut
2. 传递: `{ task: "test-execution", projectPath, scope: "重构范围", mode: "ut", context: { testDesign: { utCases } } }`
3. 记录基线: testResults (通过率 + 覆盖率)

### 保护网检查清单

```markdown
## 保护网状态

### 测试覆盖
| 文件 | 现有覆盖 | 目标覆盖 | 差距 | 操作 |
|------|---------|---------|------|------|
| fileA.js | 0% | 70% | 需全补 | 新增测试 |
| fileB.js | 45% | 70% | 需补充 | 补边界用例 |
| fileC.js | 80% | 70% | 达标 | 无需操作 |

### 关键路径测试
- [ ] 正常输入→正常输出
- [ ] 边界条件覆盖
- [ ] 异常处理覆盖
- [ ] 外部依赖Mock正确

### 基线记录
- 测试总数: X
- 通过率: 100%
- 覆盖率: XX%
- 基线时间: YYYY-MM-DD HH:MM
```

### 决策点

- **基线全部通过** → 进入阶段3
- **基线不通过** → 说明当前代码有bug，先修复再重构

---

## 阶段3: 逐步重构

### 主Agent动作

1. 启动子Agent: `refactoring`
2. 传递: `{ task: "refactoring", projectPath, scope, context: { refactorPlan, baselineTests, baselineCoverage } }`

### 重构检查清单（子Agent使用）

每步重构前检查:
- [ ] 这一步改动 ≤10 行？
- [ ] 只改了一个关注点（单一职责）？
- [ ] 公共API签名不变？
- [ ] 测试仍能覆盖这段代码？

每步重构后检查:
- [ ] 运行测试 → 全部通过？
- [ ] 测试通过率不变（没少测）？
- [ ] 提交信息符合格式？

### 代码坏味道检测

重构时应主动识别并报告:

| 坏味道 | 表现 | 建议手法 |
|--------|------|---------|
| 长函数 | >50行 | 提取函数 |
| 重复代码 | 3处以上相似 | 提取公共函数 |
| 魔法数字 | 硬编码常量 | 提取常量 |
| 长参数列表 | >4个参数 | 参数对象 |
| 深层嵌套 | >3层if/for | 提前return/提取 |
| 数据泥团 | 3+字段总一起出现 | 提取对象 |
| 过度注释 | 注释解释what | 提取函数，注释说why |

---

## 阶段4: 验收

### 主Agent动作

```markdown
# 重构验收报告

## 重构目标: [目标]

## 改动统计
| 指标 | 重构前 | 重构后 | 变化 |
|------|--------|--------|------|
| 文件数 | X | X | 0 |
| 总行数 | X | Y | -Z |
| 提交次数 | - | N | - |

## 测试对比
| 指标 | 重构前 | 重构后 |
|------|--------|--------|
| 测试数 | X | X (不变) |
| 覆盖率 | XX% | XX% |
| 通过率 | 100% | 100% |

## 验收清单
- [ ] 重构范围内功能正常
- [ ] 无功能退化
- [ ] 覆盖率不降低
- [ ] 代码行数减少（或质量提升）
- [ ] 无不必要的抽象

## 状态: ✅ 重构完成
```
