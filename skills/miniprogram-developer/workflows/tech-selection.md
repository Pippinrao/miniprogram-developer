# 技术选型工作流

帮助开发者选择合适的框架和技术方案。基于多维度对比分析，输出结构化选型建议。

## 触发条件

| 关键词 |
|--------|
| "如何选择跨端框架"、"选型"、"技术选型"、"用什么框架"、"跨端对比"、"原生还是跨端" |

## 流程

```
用户: "我想做一个小程序，用什么框架好？"
  │
  ├─ ► 阶段1: 需求收集 [主Agent]
  │     收集: 目标平台、团队技术栈、性能要求、原生能力需求、项目周期、团队规模
  │     输出: 需求画像 JSON
  │     【用户确认需求画像】
  │
  ├─ ► 阶段2: 方案对比 [主Agent]
  │     执行: search_docs.py "分包加载 多端 跨平台 原生能力" --top 8
  │     加载: 命中的官方文档 + 社区共识
  │     输出: 多维度对比矩阵
  │     【用户确认对比结果，可调整权重】
  │
  └─ ► 阶段3: 输出建议 [主Agent]
        最终推荐 + 备选方案 + 迁移路径 + 风险提示
```

## 阶段1: 需求收集

### 需求问卷

向用户逐一确认以下维度（可跳过用户已知的项）：

| 维度 | 问题 | 选项 |
|------|------|------|
| 目标平台 | 需要发布到哪些平台？ | 微信小程序 / 支付宝小程序 / 百度小程序 / 抖音小程序 / H5 / App(iOS+Android) / 快应用 |
| 团队技术栈 | 团队最熟悉的技术？ | React / Vue / 原生小程序 / 无前端经验 |
| 性能敏感度 | 对性能的要求？ | 极高(游戏/动画) / 较高(流畅交互) / 一般(工具类) |
| 原生能力 | 是否需要深度原生能力？ | 需要(蓝牙/NFC/AR) / 部分需要(支付/定位) / 不需要 |
| 项目周期 | 预期开发周期？ | 紧急(1-2周) / 正常(1-3月) / 充裕(3月+) |
| 团队规模 | 开发团队规模？ | 个人 / 小团队(2-5人) / 大团队(5人+) |
| 插件需求 | 是否需要开发/使用小程序插件？ | 需要开发插件 / 需要使用第三方插件 / 不需要 |
| 长期维护 | 项目预期维护周期？ | 短期项目 / 长期维护(1年+) |

### 需求画像输出

```json
{
  "platforms": ["wechat", "h5"],
  "techStack": "react",
  "performance": "high",
  "nativeCapability": "partial",
  "timeline": "normal",
  "teamSize": "small",
  "pluginRequired": false,
  "maintenance": "long"
}
```

## 阶段2: 多维度对比

### 候选方案池

根据需求画像自动筛选候选方案：

| 需求特征 | 候选方案 |
|----------|---------|
| 仅微信小程序 | 原生 / WeUI / TDesign |
| 微信+H5 | Uni-app(Vue) / Taro(React/Vue) |
| 微信+H5+App | Uni-app / Taro / Flutter |
| 微信+支付宝+百度+抖音 | Uni-app / Taro |
| 需要开发插件 | 原生小程序（唯一选择） |
| React 技术栈 | Taro / Remax |
| Vue 技术栈 | Uni-app |
| 极高性能要求 | 原生 / Skyline 渲染引擎 |
| 个人/小团队 | Uni-app（DCloud生态完善） |

### 对比矩阵（示例）

| 维度 | 权重 | 原生小程序 | Uni-app | Taro | Flutter |
|------|------|-----------|---------|------|---------|
| 多端支持 | 高 | ★★☆ | ★★★ | ★★★ | ★★★ |
| 性能 | 高 | ★★★ | ★★☆ | ★★☆ | ★★★ |
| 学习成本 | 中 | ★★★ | ★★☆ | ★★☆ | ★☆☆ |
| 社区活跃度 | 中 | ★★★ | ★★★ | ★★★ | ★★★ |
| 原生能力 | 高 | ★★★ | ★★☆ | ★★☆ | ★★★ |
| 插件生态 | 中 | ★★★ | ★★★ | ★★★ | ★★☆ |
| 微信新特性跟进 | 高 | ★★★ | ★☆☆ | ★☆☆ | ★☆☆ |
| 招聘难度 | 中 | ★★☆ | ★★★ | ★★★ | ★★☆ |

> 实际评分根据 search_docs.py 搜索结果和用户需求画像动态调整

### 决策规则

```
优先级排序:
1. 如果需要开发插件 → 原生小程序（唯一选择，无替代方案）
2. 如果性能要求极高 + 仅微信 → 原生 + Skyline
3. 如果多端 + React → Taro
4. 如果多端 + Vue → Uni-app
5. 如果仅微信 + 快速开发 → TDesign/WeUI + 原生
6. 如果多端 + 高性能 + 大团队 → Flutter
```

## 错误处理

| 场景 | 处理 |
|------|------|
| search_docs.py 搜索无结果 | 使用静态对比矩阵 + 标注"建议查阅对应官网获取最新信息" |
| 用户需求跨越多个不兼容场景 | 列出各场景独立推荐 + 建议分项目实现 |
| 用户拒绝所有推荐方案 | 询问是否调整需求权重，重新进入阶段1 |
| 用户需求不在已知场景中 | 扩展 search_docs.py 搜索范围，咨询社区，标注"信息有限" |

## 约束
- 只提供建议，不创建项目代码
- 所有推荐标注信息来源（官方文档 URL 或社区共识）
- 对比矩阵的评分必须有依据（官方文档原话或社区统计数据）
- 如实标注信息不确定的部分
- 对于"微信新特性跟进速度"等时效性信息，提示用户自行验证
