# 热修复工作流

线上紧急问题的快速修复流程。速度优先，事后补课。

## 流程

```
用户: "线上挂了" / "紧急修复"
  │
  ├─ ► 阶段1: 紧急评估 [主Agent]
  │     评估影响范围 → 选择修复策略 → 决策
  │     策略: 回滚(5min) / 热修复(30min) / 配置切换(10min)
  │     决策: 选策略 → 阶段2
  │
  ├─ ► 阶段2: 定位+修复
  │     步骤1 [子Agent: debugging]: 定位根因 → 输出 fixSuggestion
  │     步骤2 [子Agent: code-implementation]: 按 fixSuggestion 最小改动修复
  │     决策: 编译通过 → 阶段3
  │
  └─ ► 阶段3: 灰度上线 + 监控 + 事后 [主Agent]
        灰度发布 → 监控观察 → 全量/回滚
        事后: 补充UT + 加告警 + 复盘
```

---

## 阶段1: 紧急评估

### 评估模板

```markdown
# 紧急评估

## 影响范围
- 影响用户: [全部 / X% / 部分区域]
- 影响功能: [登录/支付/核心流程]
- 开始时间: [什么时候开始的]
- 最近变更: [最近一次发版时间/内容]
- 当前状态: [ ]服务不可用 [ ]功能异常 [ ]数据错误

## 根因初步判断
- 是否与最近发版相关？ [是/否/不确定]
- 错误日志关键信息: [有/无，具体内容]

## 策略选择

> **微信小程序发布机制说明**: 代码变更版本必须通过微信审核，审核时间通常数小时到数天。不存在"跳过审核"的紧急通道。"热修复"的代码改动仍须走正常上传→提审流程。以下灰度流程假定审核已通过。

| 策略 | 条件 | 耗时 | 风险 | 适用 |
|------|------|------|------|------|
| 回滚 | 最近24h内有发版 | 分钟级 | 低 | 部署引起的问题 |
| 配置切换 | 有备用服务/开关 | 分钟级 | 低 | API/配置问题 |
| 代码热修复 | 确定是代码bug | 小时/天级（含审核） | 中 | 逻辑错误 |
```

### 策略决策树

```
最近24h内有发版？
  ├─ 是 → 回滚（首选，最快最安全）
  │         └─ 回滚后问题解决？ → 是 → 记录，复盘
  │         └─ 回滚后问题仍在？ → 不是部署问题 → 热修复
  │
  └─ 否 → 有配置开关/备用服务？
            ├─ 是 → 配置切换
            └─ 否 → 代码热修复
```

## 阶段2: 快速修复

### 主Agent动作

步骤1: 定位根因 [子Agent: debugging]
1. 启动子Agent: `debugging`
2. 传递: `{ task: "debugging", projectPath, scope, context: { errorInfo: "问题描述" } }`
3. 从输出中获取 `rootCause` 和 `fixSuggestion`

步骤2: 快速修复 [子Agent: code-implementation, 热修复模式]
4. 启动子Agent: `code-implementation` (热修复模式)
5. 传递: `{ task: "code-implementation", projectPath, scope, context: { hotfix: true, description, fixSuggestion } }`
3. **关键提醒**: 子Agent在热修复模式下：
   - 不做完整TDD
   - 不写新测试（事后补充）
   - 不重构相关代码
   - 只验证编译通过
   - 改动控制在最小范围

### 子Agent执行约束

```
热修复模式 (code-implementation)：
✅ 按 fixSuggestion 最小改动修复
✅ 本地编译检查
✅ 最小改动修复（1-3行，最多1个文件）
✅ 本地编译检查
✅ git commit (前缀: hotfix:)
❌ 不写测试
❌ 不重构
❌ 不改无关代码
❌ 不更新依赖
```

## 阶段3: 灰度上线 + 监控 + 事后

### 3.1 灰度发布流程

```
1. 上传代码: 通过微信开发者工具或 miniprogram-ci 上传（`miniprogram-ci upload`）
2. 提交审核时选择"分阶段发布"（灰度发布）
3. 灰度比例（微信支持的标准选项）:
   - 先 1%，观察
   - 扩大到 5%、10%、20%、30%、50%
   - 全量 100%
   注: 具体可选比例和放量节奏以微信后台实际功能为准。

每一步观察指标:
  - 错误率是否下降
  - 核心功能是否恢复
  - 是否有新错误出现
```

### 3.2 监控检查清单

```markdown
## 灰度监控清单

### 1% 阶段 (5min)
- [ ] 错误率: 修复前 ___% → 当前 ___%
- [ ] 核心功能可用: [是/否]
- [ ] 无新增错误类型: [是/否]
- [ ] 用户反馈: [正常/有异常]

### 5% 阶段 (5min)
- [ ] 错误率持续下降: [是/否]
- [ ] 无性能退化: [是/否]
- [ ] API响应时间正常: [是/否]

### 25% 阶段 (10min)
- [ ] 错误率恢复到正常水平: [是/否]
- [ ] 所有功能正常: [是/否]
- [ ] 决定: 继续全量 / 回滚

### 全量后 (持续30min)
- [ ] 错误率稳定: [是/否]
- [ ] 用户投诉量正常: [是/否]
```

### 3.3 异常回滚决策

任一阶段出现以下情况，**立即停止灰度，回滚**:
- 错误率不降反升
- 出现新的严重错误
- 核心功能仍不可用
- 性能明显退化

## 与Bugfix的区别

| | Bugfix | Hotfix |
|---|--------|--------|
| 触发 | 用户报告/测试发现 | 线上监控/大量投诉 |
| 速度要求 | 正常节奏 | 尽快（分钟级） |
| 测试策略 | 完整UT+回归 | 编过即可，事后补 |
| 发布流程 | 正常CI/CD | 灰度发布 |
| 改动范围 | 可以适当重构 | 最小改动（1-3行） |
| 审批流程 | 正常PR+Review | 紧急通道 |
| 事后要求 | 验证修复即可 | 必须复盘+补测+加监控 |

---

## 错误处理

| 场景 | 处理 |
|------|------|
| debugging 无法在15分钟内定位根因 | 切换策略：回滚最近部署 → 评估是否降级（切到备用方案）→ 扩大排查团队 |
| 修复后灰度1%阶段错误率上升 | 立即回滚灰度，检查修复是否引入新问题，重新审查修复代码 |
| 编译/构建失败（紧急压力下更易出错） | 检查语法和依赖后重试；持续失败则评估是否可以配置切换代替代码修复 |
| 灰度过程中发现新问题（非目标bug） | 评估严重度：核心功能不可用→回滚；非核心→记录后继续灰度 |
| 用户/监控反馈修复无效（问题依旧） | 回滚灰度，扩大scope重新debugging，考虑是否根因定位错误 |
