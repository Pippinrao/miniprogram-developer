# 知识查询工作流

纯文档查询，无代码操作。主Agent直接处理，不启动子Agent。

## 触发条件

| 关键词 |
|--------|
| "如何使用XXX"、"XXX怎么用"、"查XXX文档" |
| "XXX的API"、"XXX是什么"、"帮我了解XXX" |
| "查询XXX"、"XXX有哪些限制"、"XXX的最佳实践" |

## 流程

```
用户: "如何在小程序中使用蓝牙"
  │
  ├─ ► 步骤1: 提取搜索关键词 [主Agent]
  │     从用户问题中提取2-3个精准关键词
  │     例: "如何使用蓝牙" → "蓝牙 BLE 设备连接"
  │     如需求复杂（涉及多个领域），提取多组关键词分别搜索
  │
  ├─ ► 步骤2: 语义搜索 [主Agent]
  │     执行: python tools/search_docs.py "<关键词>" --top 5
  │     如涉及设计规范: 添加 --category design
  │     【检查搜索结果质量】
  │
  ├─ ► 步骤3: 搜索结果评估 [主Agent]
  │     ├─ 最高得分 ≥ 0.5: 直接进入步骤4
  │     ├─ 最高得分 0.3-0.5: 标注"信息有限"，建议用户查阅官网
  │     ├─ 最高得分 < 0.3: 同义词重试搜索 → 查 FRAMEWORK_INDEX → 告知用户
  │     └─ search_docs.py 失败: 降级为 FRAMEWORK_INDEX.md 手动查找
  │
  ├─ ► 步骤4: 加载文档 [主Agent]
  │     Read 最相关的2-3个 .md 文件，提取关键信息
  │
  └─ ► 步骤5: 输出回答 [主Agent]
        结构化输出:
        - 问题直接答案
        - 关键API/配置/限制
        - 代码示例（如果文档包含）
        - 出处: 引用具体的官方文档路径和URL
        如信息不完整，标注"建议查阅官网获取最新信息"
        【询问用户是否满意，是否需要进一步搜索】
```

## 搜索技巧

- 用完整的中文短语，避免单个英文缩写
- 如结果不理想，用同义词重新搜索（最多3轮）
- 涉及多方面的查询（如"蓝牙+音频"），分别搜索后合并结果
- category 过滤: framework 用于框架/API，design 用于UI规范，api 用于具体API，component 用于组件
- 搜索结果可按 `--json` 输出以获得更精确的匹配度数据

## 输出格式

```markdown
## 回答: [用户问题]

[直接答案，从文档提取，标注出处]

## 关键信息

- API/方法: [xxx] — 出处: framework/xxx.md
- 配置项: [xxx] — 出处: framework/xxx.md
- 限制/注意: [xxx]

## 信息完整度

[完整 / 部分 / 有限] — [说明缺失的方面]

## 相关文档

- [标题](路径) — 匹配度: 0.XX
- ...
```

## 错误处理

| 场景 | 处理 |
|------|------|
| search_docs.py 执行失败（Python环境/依赖问题） | 降级到 FRAMEWORK_INDEX.md 手动查找；告知用户搜索工具不可用 |
| 搜索结果为空（0条） | 调整关键词重试（最多3轮）；仍为空则告知用户并建议查阅官网搜索 |
| 目标文档文件不存在（路径引用过期） | 跳过该文件，使用次优结果；告知用户文档可能已更新 |
| 所有结果得分 < 0.3 | 标注"信息有限"，建议查阅 develop.weixin.qq.com 官方搜索 |
| 多文档信息冲突 | 优先信任最新版本（按爬取时间判断）；标注冲突并展示两条信息 |
| 用户对答案不满意 | 进入反馈循环：询问缺失的方面 → 调整关键词 → 补充搜索（最多2轮反馈） |
| 用户问题过于宽泛（如"介绍小程序框架"） | 引导缩小范围：询问具体想了解框架的哪个方面 |

## 约束

- 只查文档不写代码
- 所有回答必须标注官方文档出处
- 搜索结果不理想时主动告知用户并建议查阅官网
- 不要杜撰不存在的API或参数
- 信息不确定时标注"需查阅最新官方文档确认"
- 涉及时效性信息（版本要求、限制数据）时提示用户以官方最新文档为准
