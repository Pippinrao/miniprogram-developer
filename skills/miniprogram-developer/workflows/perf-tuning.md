# 性能优化工作流

系统化的性能诊断与优化流程。聚焦 setData 调用、包体积、首屏渲染三大核心维度，按优先级逐个修复。

## 流程

```
用户: "性能优化" / "小程序太慢了" / "包体积超标"
  │
  ├─ ► 阶段1: 性能诊断 [子Agent: code-analysis]
  │     聚焦: setData 频率 + 包体积 + 首屏渲染
  │     参考: 官方运行时性能、setData、包体积优化文档
  │     输出: 优先级排序的性能问题清单
  │     决策: 用户确认优化范围 → 阶段2
  │
  ├─ ► 阶段2: 逐一修复 [子Agent: code-implementation, mode=fix]
  │     按优先级从高到低, 每轮: 修复 → 编译验证 → 下一项
  │     决策: 全部修复完成 → 阶段3
  │
  └─ ► 阶段3: 验收 [主Agent]
        before/after 对比: 包体积估算、setData 次数、首屏耗时
       输出: 性能优化报告 + 检查清单
```

---

## 阶段1: 性能诊断

### 主Agent动作

1. 理解用户诉求，确定优化重点:
   - 用户明确提到卡顿/慢 → 优先 setData + 首屏
   - 用户提到包体积/分包 → 优先包体积
   - 未明确 → 全维度诊断

2. 启动子Agent: `code-analysis`
3. 传递参数:

```
task: "code-analysis"
projectPath: [项目路径]
scope: [项目 pages/ + components/ + app.js + app.json]
reference: [
  "official-docs/framework/运行时性能.md",
  "official-docs/framework/合理使用_setData.md",
  "official-docs/framework/代码包体积优化.md"
]
context: {
  focus: "performance",
  dimensions: ["setData", "packageSize", "firstScreen"]
}
```

4. 子Agent 分析完成后，主Agent 将其输出整理为优先级排序的问题清单:

```markdown
## 性能诊断结果

### setData 调用分析
| 优先级 | 文件 | 问题 | 当前值 | 建议值 | 预计收益 |
|--------|------|------|--------|--------|---------|
| P0 | pages/xxx/xxx.js | setData 在循环中调用 | N次/帧 | 合并为1次 | 减少 N-1 次渲染 |
| P1 | pages/yyy/yyy.js | 单次 setData 数据量过大 | ~200KB | <100KB | 减少通信耗时 |
| P2 | components/zzz.js | 高频 setData (scroll 事件) | 每次滚动 | 节流 100ms | 减少无效渲染 |

### 包体积分析
| 优先级 | 问题 | 当前状态 | 建议 | 预计减少 |
|--------|------|---------|------|---------|
| P0 | 主包包含大图片 (>200KB) | /images/bg.png 350KB | 迁移CDN或压缩 | -300KB |
| P1 | 未使用组件未移除 | components/old-xxx/ | 删除 | -XX KB |
| P2 | 分包配置不合理 | 主包过大 | 拆分独立功能到分包 | 主包 -XX KB |

### 首屏渲染分析
| 优先级 | 文件 | 问题 | 建议 |
|--------|------|------|------|
| P0 | pages/index/index.js | onLoad 同步请求阻塞 | 异步加载+骨架屏 |
| P1 | pages/index/index.wxml | 首屏DOM节点过多 | 懒加载非首屏内容 |
| P2 | app.js | 启动时加载不必要插件 | 按需加载 |
```

5. **必须等用户确认**优化范围和优先级后才能进入阶段2。用户可以:
   - 接受全部建议
   - 仅选择 P0/P1 项
   - 排除特定文件或修改项

---

## 阶段2: 逐一修复

### 主Agent动作

1. 按优先级从高到低，逐项启动修复:

```
# 对每一项修复任务:
1. 启动子Agent: code-implementation (修复模式)
2. 传递: {
    task: "code-implementation",
    projectPath: [项目路径],
    scope: [当前修复项涉及的文件],
    context: {
      fixSuggestion: [阶段1中该问题的修复建议],
      mode: "fix"
    }
  }
3. 等待子Agent 完成修复
4. 验证: 检查编译是否通过 (检查子Agent 返回的 testResults)
5. 确认 → 进入下一项修复
```

2. 每项修复的子Agent执行流程:

```
修改代码 → 编译验证 → 通过
                        ↓
                     下一项
                        ↓
                     不通过
                        ↓
                  回滚 → 记录 → 跳过
```

3. 修复过程中遵循的约束:
   - 每项修复独立，避免交叉影响
   - 修改后必须通过编译验证
   - 编译不通过则回滚该次修改，记录并跳过
   - 不引入新的性能问题

4. 如某项修复涉及多个文件（如分包拆分），一次性修复该问题的所有关联文件。

### 修复进度追踪

主Agent维护修复进度表:

```markdown
## 修复进度

| # | 优先级 | 问题 | 文件 | 状态 | 备注 |
|---|--------|------|------|------|------|
| 1 | P0 | setData 循环调用 | pages/xxx/xxx.js | ✅ 已修复 | 合并3次调用为1次 |
| 2 | P0 | 主包大图片 | /images/bg.png | ✅ 已迁移 | 迁移至CDN, -300KB |
| 3 | P1 | 未使用组件 | components/old-xxx/ | ⬜ 待修复 | |
| 4 | P1 | setData 数据量过大 | pages/yyy/yyy.js | ⬜ 待修复 | |
| 5 | P2 | scroll 高频 setData | components/zzz.js | ⬜ 待修复 | |
```

## 性能检查清单

优化完成后逐项自检:

- [ ] **setData 每次数据量 < 100KB** — 单次 setData 传递的数据量控制在 100KB 以内，避免跨线程通信成为瓶颈
- [ ] **避免频繁 setData (合并多次调用)** — 同一帧内的多次 setData 合并为一次调用；scroll/resize 等高频事件使用节流 (100~200ms)
- [ ] **首屏关键路径减少同步操作** — onLoad/onShow 中避免同步等待；使用异步加载 + 骨架屏/loading 态提升感知速度
- [ ] **图片使用 webp 格式 + 懒加载** — 图片资源优先使用 webp 格式；非首屏图片使用 lazy-load 属性延迟加载
- [ ] **分包加载配置合理** — 独立功能模块配置为分包；主包仅保留首屏必需资源；预加载规则与实际用户路径匹配
- [ ] **不使用的组件/页面已移除** — 清理 app.json 中未引用的页面路径；清理 components/ 中未被使用的组件目录

### 常见性能问题及修复对照

| 问题现象 | 诊断方法 | 修复手段 |
|---------|---------|---------|
| 页面卡顿/操作延迟 | 检查 setData 调用频率和数据量 | 合并 setData、数据 diff 后再 setData |
| 列表滚动掉帧 | 检查 scroll 事件中的 setData | 节流/防抖 + 只更新变化项 |
| 首屏白屏时间长 | 检查 onLoad 同步操作和组件数量 | 异步加载 + 骨架屏 + 减少首屏组件 |
| 主包体积超标 | 检查主包文件大小分布 | 图片迁移 CDN、非核心功能移至分包 |
| 分包加载失败 | 检查 preloadRule 配置 | 调整预加载规则、检查分包路径 |
| 页面切换慢 | 检查页面 onLoad 初始化 | 减少初始化逻辑、预加载数据 |
| 内存占用高 | 检查未释放的定时器/监听 | onHide/onUnload 中清理资源 |

## 错误处理

| 场景 | 处理 |
|------|------|
| code-analysis 性能诊断无结果（分析工具无法量化性能） | 基于静态代码规则检查（setData频率/包体积/组件数）输出经验性诊断 |
| 逐项修复后编译失败 | 回滚该项修改，记录失败原因，跳过该优化项继续下一项 |
| 修复后性能无改善或退化 | 回滚修改，标注"无效优化"，记录原因供后续分析 |
| before/after 对比数据相差过大（疑似测量误差） | 重新测量3次取平均值，仍异常则标注"测量不稳定" |
| 用户拒绝全部优化建议 | 输出诊断报告供用户自行参考，标记"用户延后处理" |
| 包体积优化导致功能异常（分包/图片迁移后 404） | 测试所有受影响页面的资源加载，异常则回滚 |
