# 技能盲审工作流 — skill-audit

## 触发

用户说"盲审"、"审查skill"、"audit"、"评分"、"skill质量"

## 流程

```
用户: "帮我盲审一下 harmonyos-dev"
  │
  ├─ ► 阶段1: 确定审查范围 [主Agent]
  │     1. 确认审查范围（全量/增量/指定维度）
  │     2. 确认目标分数和最大迭代轮次
  │     3. 加载 `patterns/blind-review-cycle.md` + `patterns/quality-gates.md`
  │     决策: 范围确认 → 阶段2
  │
  ├─ ► 阶段2: 并行盲审 [10+ 子Agent]
  │     并行启动多个无上下文评分Agent:
  │     - Agent A: SKILL.md 主调度文件
  │     - Agent B: 所有Agent定义文件
  │     - Agent C: 工作流文件（前半）
  │     - Agent D: 工作流文件（后半）
  │     - Agent E: 参考文档
  │     - Agent F: 协议文件
  │     - Agent G: 官方文档/知识库
  │     - Agent H: 真实用户视角
  │     - Agent I: 工具链
  │     - Agent J: 交叉引用一致性
  │     - Agent K: API准确性专项
  │     - Agent L: 设计/规范文档
  │     等待全部返回 → 汇总评分
  │
  └─ ► 阶段3: 评分汇总 [主Agent]
        1. 计算各维度加权平均分
        2. 汇总所有 Critical/Major/Minor 问题
        3. 输出结构化审查报告
        4. 判定是否通过质量门禁
```

---

## 阶段1: 确定审查范围

### 审查范围选项

```markdown
## 审查范围确认

### 范围
- [ ] 全量审查（所有文件）
- [ ] 增量审查（仅变更文件）
- [ ] 指定维度（API准确性 / 一致性 / 完整性 / ...）

### 目标
- 目标分数: [ ] 7.0 / [ ] 8.0 / [ ] 9.0
- 最大迭代轮次: [ ] 3 / [ ] 5 / [ ] 不限
```

### 知识加载

- `patterns/blind-review-cycle.md` — 盲审循环机制
- `patterns/quality-gates.md` — 质量门禁与评分维度
- 目标技能的 `SKILL.md` — 了解技能结构

---

## 阶段2: 并行盲审

### 评分Agent设计原则

1. **无上下文**: 每个评分Agent是全新会话，不知道之前的评分结果
2. **伪装真实使用**: Agent提示词让Agent以为自己在实际使用该技能
3. **并行最大化**: 所有评分Agent同时启动
4. **覆盖全维度**: 不同Agent聚焦不同维度/文件

### 评分Agent提示词模板

```
你是一个 [领域] 开发者，正在评估 "[技能名]" 技能。
你需要决定是否将其用于你的 [具体项目描述]。

**重要**: 你不是在扫描问题 — 想象你自己正在使用每个组件进行实际开发。

请阅读 [文件列表] 并按照以下维度评分（0-10）:
1. 正确性: API/命令/语法是否准确
2. 完整性: 是否覆盖所有必要场景
3. 一致性: 交叉引用和字段名是否统一
4. 实用性: 真实开发场景是否可用
5. 清晰度: 是否易于理解
6. 错误处理: 失败场景是否有应对

对每个维度给出一个分数和一句话理由。
最后给出综合评分。
```

### 并行启动

```markdown
主Agent同时启动以下评分子Agent:

| Agent | 目标文件 | 评分重点 |
|-------|---------|---------|
| A | SKILL.md, README.md | 架构合理性、角色清晰度 |
| B | agents/*.md (全部) | Agent定义质量、输入输出规范 |
| C | workflows/*.md (前半6个) | 流程逻辑、阶段设计 |
| D | workflows/*.md (后半6个) | 流程逻辑、阶段设计 |
| E | references/*.md (全部) | API准确性、代码质量、实用性 |
| F | protocols/context-exchange.md | 字段完备性、一致性 |
| G | official-docs/ | 文档真实性、页数准确性 |
| H | 模仿真实开发者使用 | 用户体验、上手难度 |
| I | tools/ | 工具功能正确性 |
| J | 全技能交叉引用 | 路径验证、字段一致性 |
| K | 全技能API引用 | 每个API的真实性验证 |
| L | design/ 规范文档 | 内容真实性 |
```

### 决策点

- 全部Agent正常返回 → 阶段3
- 部分Agent超时/失败 → 使用已返回的结果，标注缺失维度
- 全部失败 → 减小审查范围重试

## 阶段3: 评分汇总

### 主Agent动作

1. 收集所有评分Agent返回的 `keyFindings`
2. 按维度计算加权平均分
3. 按严重度分类所有问题
4. 生成结构化审查报告

## 错误处理

| 场景 | 处理 |
|------|------|
| 目标技能目录不存在 | 确认路径，让用户提供正确路径 |
| 技能文件量过大（>50个文件） | 建议分批审查，或仅审查关键模块 |
| 评分Agent返回非标准格式 | 提取可用的分数和发现，标注缺失 |
| 多次评分差异过大（>2分） | 增加评分Agent数量，取中位数 |
| 搜索结果冲突（Agent A说API不存在，Agent B说存在） | 用官方文档做最终裁定 |
