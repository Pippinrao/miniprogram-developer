# 技能迭代改进工作流 — skill-improve

## 触发

用户说"改进skill"、"修复问题"、"提升分数"、"优化skill"

## 流程

```
用户: "帮我改进 harmonyos-dev 的评分"
  │
  ├─ ► 阶段1: 读取审查报告 [主Agent]
  │     1. 加载最新的盲审报告
  │     2. 按严重度+影响+修复成本排序问题
  │     3. 制定本轮修复计划
  │     决策: 用户确认修复范围 → 阶段2
  │
  ├─ ► 阶段2: 批量修复 [子Agent执行，主Agent调度]
  │     主Agent对每项修复启动子Agent，传入问题描述和修复方案。
  │     子Agent执行: 读取问题文件 → 应用修复 → 验证 → 返回结果。
  │     主Agent只传文件路径和修复方案，不自己读/写文件。
  │     按优先级从高到低逐项启动子Agent修复。
  │     决策: 所有计划项修复完成 → 阶段3
  │
  ├─ ► 阶段3: 验证修复 [主Agent]
  │     1. 逐项检查修复是否正确
  │     2. 确保无回归问题
  │     决策: 验证通过 → 阶段4
  │
  └─ ► 阶段4: 重新盲审 [主Agent → 启动 skill-audit]
        1. 启动新的盲审（增量审查）
        2. 对比前后分数
        3. 达标 (≥9.0) → 输出完成报告
        4. 未达标 → 返回阶段1继续迭代
```

---

## 阶段1: 读取审查报告

### 主Agent动作

1. 找到最新的审查报告（如 `IMPROVEMENT_PLAN.md` 或最新的 audit 输出）
2. 按优先级矩阵排序问题：

### 优先级矩阵

```
影响 × 修复成本 = 优先级

影响高 + 成本低 → P0 (Critical，立即修复)
影响高 + 成本高 → P1 (Major，本轮修复)
影响低 + 成本低 → P2 (Minor，批量修复)
影响低 + 成本高 → P3 (Info，记录待办)
```

### 修复计划模板

```markdown
## 改进计划 — Round N

### 目标分数: 9.0 | 当前分数: X.X

### P0 — Critical (立即修复)
| # | 问题 | 文件 | 预计改动 | 修复方式 |
|---|------|------|---------|---------|
| 1 | | | | |

### P1 — Major (本轮修复)
| # | 问题 | 文件 | 预计改动 | 修复方式 |
|---|------|------|---------|---------|
| 1 | | | | |

### P2 — Minor (时间允许)
| # | 问题 | 文件 | 修复方式 |
|---|------|------|---------|

### 预计提升: X.X → Y.Y
```

## 阶段2: 批量修复

### 修复原则

1. **单一职责**: 每次只修复一个问题
2. **先验证后修复**: 读文件确认问题确实存在
3. **最小改动**: 只改需要的行，不顺手重构
4. **可回溯**: 修复后可单独回滚

### 常见修复模式的执行

#### 修复1: 字段名不一致
```
问题: Agent输出用 filesCreated，协议要求 filesChanged
修复: 将Agent输出的 filesCreated → filesChanged
      (或在协议中添加 filesCreated 作为接受的别名)
```

#### 修复2: 捏造API
```
问题: 代码中使用了不存在的 element.clear()
修复: 替换为 element.input('')
      添加注释说明替代方案
```

#### 修复3: 阶段顺序错误
```
问题: 测试设计在代码实现之后 → 不是TDD
修复: 调整阶段顺序: 需求设计 → 测试设计 → 代码实现
      或添加说明: UT在实现阶段内完成，此阶段专为E2E
```

#### 修复4: 安全漏洞
```
问题: catch块返回 error: err.message 暴露内部细节
修复: 改为 console.error + 通用错误消息
      检查同类模式
```

#### 修复5: 引用断裂
```
问题: 引用的文件路径不存在
修复: 创建缺失的文件或更新路径
```

### 修复进度追踪

```markdown
## 修复进度

| # | 优先级 | 问题 | 文件 | 状态 | 备注 |
|---|--------|------|------|------|------|
| 1 | P0 | | | ✅ 已修复 | |
| 2 | P0 | | | ✅ 已修复 | |
| 3 | P1 | | | ⬜ 待修复 | |
| 4 | P1 | | | ⬜ 待修复 | |
```

## 阶段3: 验证修复

### 验证检查清单

```markdown
## 修复验证

### 直接验证
- [ ] 每个修复的文件已重读确认
- [ ] 修复内容正确且无引入新错误
- [ ] 修复后的代码语法正确

### 一致性验证
- [ ] 修复不破坏其他文件的引用
- [ ] 修复后的字段名与其他文件一致
- [ ] 修复不引入新的协议不兼容

### 回归验证
- [ ] 无之前已通过的检查项变为失败
- [ ] 无新的交叉引用断裂
```

---

## 阶段4: 重新盲审

### 主Agent动作

1. 启动 `workflows/skill-audit.md` 工作流
2. 使用**增量审查**模式（仅审查变更文件+影响范围）
3. 对比前后分数:

```markdown
## 改进效果对比

| 指标 | 修复前 | 修复后 | 变化 |
|------|--------|--------|------|
| 综合评分 | X.X | Y.Y | +Z.Z |
| Critical | N | 0 | -N |
| Major | N | M | - |
| Minor | N | M | - |

### 本轮修复统计

| 指标 | 数量 |
|------|------|
| 修复文件数 | N |
| 总改动行数 | +X / -Y |
| 新增/修复 | P0:N, P1:M, P2:K |
```

### 决策点

- 评分 ≥ 9.0 → 达标，输出完成报告
- 评分提升但 < 9.0 → 返回阶段1继续迭代
- 评分未提升 → 分析原因，调整策略
- 迭代超过5轮 → 评估是否需调整目标分数

---

## 错误处理

| 场景 | 处理 |
|------|------|
| 找不到审查报告 | 先运行 `skill-audit` 生成报告 |
| 修复后引入新问题 | 回滚该修复，标注"需重新设计方案" |
| 连续3次修复同一问题失败 | 暂停，评估是否需根本性方案变更 |
| 修复后评分反而下降 | 回滚本轮所有修复，逐项排查原因 |
| 迭代5轮后仍未达标 | 评估剩余差距是否可接受，或调整目标 |
