# 质量门禁与评分维度

## 评分维度

### 维度1: 正确性 (Correctness) — 权重 25%

检查所有技术内容的准确性：
- API/方法/命令的名称和签名是否正确
- 代码示例是否可以运行
- 是否有捏造（fabricated）的内容
- 引用的文档路径是否正确

**评分标准**:
- 10: 所有API/命令经过真实环境验证
- 7: 有少量签名错误，但无捏造API
- 4: 存在捏造的API/方法
- 1: 大量错误信息

### 维度2: 完整性 (Completeness) — 权重 20%

检查是否覆盖了所有必要场景：
- 工作流是否覆盖了所有触发关键词
- Agent定义是否包含了所有必要字段
- 错误处理是否覆盖了常见失败场景
- 知识库是否涵盖核心领域

**评分标准**:
- 10: 所有场景全覆盖，无遗漏
- 7: 主要场景已覆盖，边缘场景有缺失
- 4: 部分核心场景缺失
- 1: 严重不完整

### 维度3: 一致性 (Consistency) — 权重 15%

检查内部一致性：
- 交叉引用是否正确
- 字段名是否统一
- Agent输出是否与协议对齐
- 命名规范是否一致

**评分标准**:
- 10: 所有引用正确，字段完全一致
- 7: 有少量次要不一致
- 4: 存在协议-实现不匹配
- 1: 多处断裂引用

### 维度4: 实用性 (Practicality) — 权重 20%

检查是否真的能用于开发：
- 流程是否可以直接执行
- 模板是否可以直接使用
- Token预算是否合理
- 是否考虑了真实开发中的约束

**评分标准**:
- 10: 可直接用于生产开发
- 7: 需要少量适配即可使用
- 4: 概念正确但落地困难
- 1: 纸上谈兵，无法实用

### 维度5: 清晰度 (Clarity) — 权重 10%

检查是否易于理解：
- 角色和职责是否明确
- 指令是否无歧义
- 图表和代码块是否有助于理解
- 术语是否一致

**评分标准**:
- 10: 一目了然，无理解障碍
- 7: 大部分清晰，少数地方有歧义
- 4: 多处需要猜测
- 1: 理解困难

### 维度6: 错误处理 (Error Handling) — 权重 10%

检查对失败场景的覆盖：
- 每个工作流是否有错误处理表
- 是否考虑了工具不可用的情况
- 是否有重试和超时机制
- 是否有降级策略

**评分标准**:
- 10: 所有失败场景都有应对方案
- 7: 主要失败场景已覆盖
- 4: 只有基本错误处理
- 1: 无错误处理

## 加权计算

```
综合评分 = 正确性×0.25 + 完整性×0.20 + 一致性×0.15 + 实用性×0.20 + 清晰度×0.10 + 错误处理×0.10
```

## 质量门禁

### 必须通过 (Hard Gates)

| 门禁 | 阈值 | 不通过后果 |
|------|------|-----------|
| 综合评分 | ≥ 7.0 (推荐 8.0+) | 禁止发布，继续迭代 |
| Critical 问题数 | = 0 | 必须全部修复 |
| Major 问题数 | ≤ 2 | 优先修复 |
| 交叉引用断裂数 | = 0 | 逐个修复 |
| 捏造 API 数 | = 0 | 替换为真实API |
| 虚假文档页面数 | = 0 | 重新采集 |

### 建议通过 (Soft Gates)

| 门禁 | 阈值 | 不通过时 |
|------|------|---------|
| Minor 问题数 | ≤ 5 | 建议修复 |
| 测试/示例覆盖率 | ≥ 70% | 补充 |
| 协议合规率 | 100% | 修复不兼容 |

## 改进优先级矩阵

```
影响 × 修复成本 = 优先级

影响高 + 成本低 → P0 (立即修复)
影响高 + 成本高 → P1 (规划修复)
影响低 + 成本低 → P2 (批量修复)
影响低 + 成本高 → P3 (记录待办)
```
