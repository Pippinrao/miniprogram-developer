# 盲审循环评分机制

## 核心思想

用**无上下文的子Agent**模拟第三方评审，消除"自我评分偏高"的偏差，通过迭代循环持续提升技能质量。

## 循环流程

```
┌─────────────────────────────────────────────────────┐
│              盲审循环 (Blind Review Cycle)           │
├─────────────────────────────────────────────────────┤
│                                                     │
│   1. 评分 (Score)                                   │
│      └─ 启动无上下文子Agent → 伪装真实使用场景       │
│      └─ 并行评分（10+路），覆盖所有维度              │
│                                                     │
│   2. 分析 (Analyze)                                 │
│      └─ 汇总各维度得分 → 识别薄弱环节               │
│      └─ 按严重度排序问题清单                         │
│                                                     │
│   3. 规划 (Plan)                    达标 → 停止      │
│      └─ 制定改进计划 → 优先修复Critical/Major        │
│                                                     │
│   4. 执行 (Execute)                                 │
│      └─ 批量修复 → 逐项验证                         │
│                                                     │
│   5. 验证 (Verify)                  达标 → 停止      │
│      └─ 重新盲审 → 对比前后分数                      │
│      └─ 未达标 → 返回步骤2                          │
│                                                     │
└─────────────────────────────────────────────────────┘
```

## 评分维度框架

| 维度 | 权重 | 评分要点 | 适用对象 |
|------|------|---------|---------|
| 正确性 (Correctness) | 25% | API/命令/语法准确，无捏造 | 所有文件 |
| 完整性 (Completeness) | 20% | 覆盖所有必要场景，无遗漏 | 工作流、Agent、文档 |
| 一致性 (Consistency) | 15% | 交叉引用正确，字段名统一 | 全技能 |
| 实用性 (Practicality) | 20% | 真实开发场景可用 | 工作流、参考文档 |
| 清晰度 (Clarity) | 10% | 易于理解，无歧义 | 所有文件 |
| 错误处理 (Error Handling) | 10% | 失败场景有应对方案 | 工作流、协议 |

## 评分Agent设计

### 关键技巧

1. **无上下文**: 每个评分Agent都是全新会话，不知道之前的评分结果
2. **伪装真实使用**: Agent提示词中要让Agent以为自己在实际使用该技能开发
3. **并行最大化**: 评分任务无依赖，尽可能并行启动（理想10+路；实际取决于工具平台并行能力）
4. **覆盖全维度**: 不同Agent聚焦不同维度/文件，避免单一视角

### 提示词模板

```
你是一个 [领域] 开发者，正在评估一个名为 "[技能名]" 的AI辅助开发技能。
你需要决定是否将其用于你的 [具体项目描述]。
**不要只是扫描问题 — 想象你自己正在使用每个组件进行实际开发。**
请阅读以下文件并评分...
```

### 评分Agent分组建议

| Agent | 聚焦范围 | 评分维度 |
|-------|---------|---------|
| Agent 1 | SKILL.md 主调度文件 | 架构、清晰度、完整性 |
| Agent 2 | 所有Agent定义文件 | 角色清晰度、输入规范、输出格式 |
| Agent 3 | 工作流文件（前半） | 流程逻辑、错误处理、可操作性 |
| Agent 4 | 工作流文件（后半） | 同上 |
| Agent 5 | 参考文档 | API准确性、代码质量、实用性 |
| Agent 6 | 协议文件 | 字段完备性、一致性 |
| Agent 7 | 官方文档/知识库 | 真实性、完整性、组织性 |
| Agent 8 | 真实用户视角 | 可用性、上手难度 |
| Agent 9 | 工具链 | 功能正确性、可用性 |
| Agent 10 | 交叉引用一致性 | 路径验证、字段一致性 |
| Agent 11 | API准确性专项 | 每个API的真实性 |
| Agent 12 | 设计/规范文档 | 内容真实性 |

## 问题严重度分级

| 级别 | 定义 | 示例 | 必须修复 |
|------|------|------|---------|
| **Critical** | 会导致运行时错误 | 捏造的API、错误的命令语法 | 是 |
| **Major** | 破坏一致性/安全性 | 字段名不一致、敏感信息泄露 | 是 |
| **Minor** | 影响可维护性 | 冗余注释、命名不规范 | 建议 |
| **Info** | 改进建议 | 可提取公共函数 | 自行决定 |

## 质量门禁

| 门禁 | 阈值 | 不通过时 |
|------|------|---------|
| 综合评分 | ≥7.0 (推荐 ≥8.0) | 继续迭代 |
| Critical 问题 | 0 | 立即修复 |
| Major 问题 | ≤2 | 优先修复 |
| 交叉引用正确率 | 100% | 逐个修复 |
| API 捏造数量 | 0 | 立即修复 |
| 虚假文档页面 | 0 | 重新采集 |

## 改进策略

### 第1轮：正确性修复
- 修复所有 API 捏造、字段不一致、路径错误
- 目标：消除 Critical 和大部分 Major

### 第2轮：完整性补充
- 补充遗漏的字段、错误处理、文档引用
- 目标：消除剩余 Major 和大部分 Minor

### 第3轮：深度优化
- 添加更丰富的示例、异常处理、边界条件
- 目标：评分达到 8.5+

### 第4轮+：精细化
- 架构优化、性能改进、用户体验提升
- 目标：评分达到 9.0+

## 常见盲审发现

| 类别 | 典型问题 | 修复方向 |
|------|---------|---------|
| 字段漂移 | Agent输出字段与协议不一致 | 统一字段名，协议中声明可选别名 |
| API捏造 | 使用了不存在的API/方法 | 替换为真实API，添加兼容性注释 |
| 阶段顺序 | TDD流程中测试设计在实现之后 | 调整阶段顺序 |
| 约束冲突 | 主Agent的约束与某工作流冲突 | 添加例外说明 |
| 引用断裂 | 引用的文件路径不存在 | 验证所有交叉引用 |
| 安全泄露 | 错误信息暴露内部细节 | 服务端日志+通用错误返回 |
