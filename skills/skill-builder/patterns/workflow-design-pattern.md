# 阶段门控工作流设计模式

## 工作流=状态机

每一个工作流都是一个有明确门控条件的阶段状态机。

```
[触发] → [阶段1] → [门控1] → [阶段2] → [门控2] → ... → [验收]
              │                │
              └─ 失败 → 返回   └─ 失败 → 返回
```

## 工作流标准结构

每个工作流文件必须包含7个段落：

### 1. 触发条件 (Trigger)
明确的触发关键词列表，用表格呈现。

### 2. 流程图 (Flow Diagram)
ASCII流程图，展示阶段 → 执行者 → 决策点 → 下一阶段。

### 3. 阶段调度指令 (Phase Instructions)
每个阶段包含：
- 主Agent动作（做什么、不做什么）
- 知识加载（该阶段需要的文档）
- 子Agent传递参数（完整的结构化JSON）
- 预期输出格式

### 4. 决策点 (Decision Points)
每个阶段结束时：
- 成功条件 → 进入下一阶段
- 失败条件 → 回退/重试/终止

### 5. 错误处理 (Error Handling)
表格形式：场景 → 处理方式。至少覆盖5个场景。

### 6. 验收模板 (Acceptance Template)
Markdown格式的完成报告，标注 ✅/❌ 状态。

### 7. 占位符说明 (如果是模板)
对模板中 `{{PLACEHOLDER}}` 的解释。

## 阶段设计原则

### 原则1: 最小依赖
每个阶段只依赖上一个阶段的**摘要**输出，不依赖完整内容。

### 原则2: 用户确认门
涉及设计/方案/范围变更的阶段，**必须**等用户确认后才能继续。

### 原则3: 快速失败
能在阶段1发现的问题，不要等到阶段3。每个阶段都应有前置校验。

### 原则4: 可回溯
任何阶段失败都能回到上一个稳定状态重新开始。

### 原则5: 可并行
独立子任务能在同一阶段并行启动。

## 常见工作流模板

### 模板A: 创建型 (Create)
```
阶段1: 信息收集 [主Agent] → 阶段2: 脚手架生成 [子Agent] → 阶段3: 验收 [主Agent]
```
适用：项目初始化、技能脚手架、配置生成

### 模板B: 开发型 (Develop)
```
阶段1: 需求设计 [子Agent] → 阶段2: 并行开发 [2个子Agent] → 阶段3: 测试 [子Agent] → 阶段4: 验收 [主Agent]
```
适用：功能开发、组件开发、服务开发

### 模板C: 修复型 (Fix)
```
阶段1: 问题定位 [子Agent] → 阶段2: 修复实现 [子Agent] → 阶段3: 验证 [主Agent]
```
适用：Bug修复、紧急修复、技术债务处理

### 模板D: 审查型 (Review)
```
阶段1: 内容分析 [子Agent] → 阶段2: 问题汇总 [主Agent] → 阶段3: 报告输出 [主Agent]
```
适用：代码审查、技能审查、质量审计

### 模板E: 优化型 (Optimize)
```
阶段1: 诊断分析 [子Agent] → 阶段2: 逐项优化 [子Agent] → 阶段3: 对比验证 [主Agent]
```
适用：性能优化、包体积优化、体验优化

### 模板F: 查询型 (Query)
```
步骤1: 提取关键词 [主Agent] → 步骤2: 语义搜索 [主Agent] → 步骤3: 加载文档 [主Agent] → 步骤4: 输出回答 [主Agent]
```
适用：知识查询、API查询、技术选型

## 决策点设计

每个决策点需要明确：
1. **判断条件**: 什么算成功？什么算失败？
2. **成功路径**: 进入哪个阶段？
3. **失败路径**: 回退到哪里？重试几次？
4. **超时处理**: 等了多久算超时？超时后干什么？

### 决策点示例

| 阶段 | 成功条件 | 成功→ | 失败条件 | 失败→ |
|------|---------|-------|---------|-------|
| 需求设计 | designDoc 完整 + 用户确认 | 阶段2 | 用户否定/信息不足 | 重新设计 |
| 代码实现 | UT全部通过 + 覆盖率≥70% | 阶段3 | 失败≥3次 | 报告主Agent |
| E2E验证 | 全部场景通过 | 验收 | 有失败场景 | 返回阶段2修复 |
